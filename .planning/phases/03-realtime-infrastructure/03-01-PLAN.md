2. **Data structures:**
   ```typescript
   // Room ID -> Set of user IDs (for accurate participant count)
   const roomPresence = new Map<string, Set<string>>()

   // Room ID -> Map of socket ID -> PresenceData (for socket-level tracking)
   const socketData = new Map<string, Map<string, PresenceData>>()

   const HEARTBEAT_INTERVAL_MS = 30000 // 30 seconds
   ```

3. **handleUserJoin function:**
   - Add userId to roomPresence Set
   - Store socket-level data in socketData Map
   - Broadcast 'user-joined' event to room with: { userId, userName, userPhoto, timestamp }
   - Call broadcastPresence(roomId)

4. **handleUserLeave function:**
   - Remove socket from socketData
   - Check if user has other sockets in room (prevent false leave)
   - If no other sockets, remove userId from roomPresence
   - Broadcast 'user-left' event if user completely left
   - Clean up empty rooms from both Maps
   - Call broadcastPresence(roomId)

5. **broadcastPresence function:**
   - Get participant count from roomPresence Set
   - Broadcast 'presence-update' event with: { participantCount, participants: Array }

6. **Heartbeat cleanup interval:**
   - setInterval every HEARTBEAT_INTERVAL_MS
   - Iterate through socketData, check lastHeartbeat timestamp
   - Remove sockets with stale heartbeats (> 2x interval)
   - Log cleanup for monitoring

Follow RESEARCH.md Pattern 1 code example exactly.

   ```typescript
     const sockets = socketData.get(roomId)
     if (sockets?.has(socket.id)) {
       const socketInfo = sockets.get(socket.id)
       socketInfo.lastHeartbeat = Date.now()
     }
   })
   ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Presence events | Untrusted join/leave events from authenticated users |
| Presence data → Broadcast | Trusted presence updates sent to room participants |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | User identity in presence | mitigate | JWT authentication middleware verifies userId before presence events. Never trust client-reported userId. |
| T-03-02 | Tampering | Participant count manipulation | mitigate | Server-side Map is source of truth. Client cannot modify count. |
| T-03-03 | Denial of Service | Memory exhaustion from stale sockets | mitigate | Heartbeat cleanup every 30 seconds removes stale connections. Empty rooms cleaned up. |
| T-03-04 | Information Disclosure | Presence data leakage | mitigate | Presence events only broadcast within room namespace. Socket.IO isolation enforced. |
| T-03-05 | Tampering | Multiple tabs exploit | mitigate | Track by userId, not socketId. One user = one participant regardless of tab count. |
</threat_model>
