  <files>server/webrtc-server.ts</files>
  <action>
    Create server/webrtc-server.ts with the following implementation:

    1. **Worker singleton**: Create one mediasoup worker process on server startup
       - Configure with logLevel: 'warn'
       - Set RTC port range (40000-49999)
       - Handle worker 'died' event (exit process)

    2. **Room router management**: Create Map<roomId, MediaSoupRoom> to track room state
       - MediaSoupRoom interface: { router, transports, producers, consumers }
       - createRoomRouter() function with codec support (audio/opus, video/VP8/VP9/H.264)

    3. **WebRTC transport creation**: createWebRtcTransport() function
       - Listen on 0.0.0.0 with announced IP from env (PUBLIC_IP)
       - Enable UDP, TCP, SCTP (data channel for chat)
       - Prefer UDP for performance

    4. **Producer management**: Handle outgoing streams from clients
       - Store producers in Map<socketId, Producer>
       - Support audio and video producers
       - Close producers on disconnect

    5. **Consumer management**: Handle incoming streams for clients
       - Create consumers when new producers appear
       - Store consumers in Map<userId, Consumer[]>
       - Close consumers on disconnect

    6. **Transport cleanup**: Properly close resources on user disconnect
       - Close all producers for the transport
       - Close all consumers for the transport
       - Close the transport itself
       - Remove from room state

    Use the architecture pattern from RESEARCH.md §Pattern 1: mediasoup Server Architecture.

    **IMPORTANT**: Do NOT start the server in this file. Only export the setup functions. The actual Socket.IO integration happens in Task 3 (plan 04-02).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Socket.IO | Untrusted client sends transport/producer/consumer requests |
| mediasoup worker → OS | Privileged access to network ports (40000-49999) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | Transport creation | mitigate | Verify JWT token in Socket.IO middleware (already implemented in socket-server.ts) before allowing transport creation |
| T-04-02 | Tampering | RTP parameters | mitigate | Validate RTP parameters with Zod schema before passing to mediasoup |
| T-04-03 | Denial of Service | Port exhaustion | mitigate | Limit max transports per room (12 participants = 12 transports max), implement cleanup on disconnect |
| T-04-04 | Information Disclosure | Room state leakage | mitigate | Only emit new-producer events to authenticated room participants, not public |
| T-04-05 | Elevation of Privilege | Admin impersonation | accept | No admin privileges in SFU layer. Authorization handled at application layer. |
</threat_model>
